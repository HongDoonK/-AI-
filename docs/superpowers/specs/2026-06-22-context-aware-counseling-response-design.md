# 상담 Agent 맥락 기반 응답 다양화 설계

**작성일:** 2026-06-22
**상태:** 사용자 승인
**결정:** 추천 완료 후 상담 단계에 공통 `ResponsePlanner`를 도입한다.

## 1. 문제

현재 정책 상담 답변은 정책 DB에서 읽은 정확한 사실을 전달하지만, 같은 의도에서는 정해진
문장과 항목 순서를 반복하는 경향이 강하다.

주요 원인은 다음과 같다.

- `/agent/converse`의 서류, 혜택, 자격, 신청 방법 답변이 대부분 고정 템플릿으로 조립된다.
- `/chat`은 `LLM_PROVIDER=none`일 때 고정된 규칙 기반 답변을 사용한다.
- LLM을 사용하더라도 HF는 `temperature=0.0`, 로컬 모델은 `do_sample=False`여서 같은
  입력에 거의 같은 표현을 만든다.
- 두 상담 API가 답변의 초점, 구조, 설명 깊이, 후속 질문을 서로 다른 방식으로 결정한다.

단순히 temperature를 높이면 표현은 달라질 수 있지만, 같은 맥락의 일관성이 떨어지고
규칙 기반 모드에는 아무 효과가 없다. 여러 문장을 무작위로 선택하는 방식도 사용자 맥락과
무관한 변화만 만들 수 있다.

## 2. 목표

정책 사실은 그대로 유지하면서 상담 답변의 다음 요소를 대화 맥락에 맞게 변화시킨다.

- 이번 답변에서 먼저 강조할 내용
- 표시할 섹션과 섹션 순서
- 설명 깊이와 답변 길이
- 문장 도입부와 연결 방식
- 이전 답변과 중복되지 않는 후속 질문 또는 다음 행동

다양화는 무작위가 아니라 사용자 질문, 선택 정책, 사용자 조건, 이전 대화에 의해
결정되어야 한다.

## 3. 범위

### 적용 대상

- `POST /chat`
  - 사용자가 이미 선택한 정책에 대한 후속 상담
- `POST /agent/converse`
  - 추천 목록에서 정책을 선택한 이후의 서류, 혜택, 자격, 신청 방법 상담

### 적용 제외

- `POST /recommend`
- 정책 검색, 후보 필터링, 순위 결정
- 추천 카드의 추천 사유 및 요약 생성
- DB 로딩, 정책 사실 추출, 지원금 계산, 적격성 판정
- 정책을 선택하기 전의 추천 유도 및 정책 선택 안내

따라서 이 변경은 추천 결과의 정책 목록, 순위, 금액, 기간, 자격, 링크를 바꾸지 않는다.

## 4. 설계 원칙

1. **사실과 표현을 분리한다.** 금액, 기간, 자격, 제출 서류, URL은 기존 DB·RAG·규칙
   결과만 사용한다.
2. **맥락으로 다양화한다.** 난수나 단순 템플릿 회전 대신 현재 의도와 대화 이력을 사용한다.
3. **같은 맥락에서는 일관성을 유지한다.** 동일한 정책, 의도, 사용자 조건, 대화 단계라면
   같은 응답 계획이 선택되어야 한다.
4. **맥락이 바뀌면 설명도 바뀐다.** 사용자가 추가 정보를 제공하거나 다른 항목을 물으면
   초점, 섹션 순서, 설명 깊이, 후속 질문을 조정한다.
5. **LLM 없이 완전히 동작한다.** `LLM_PROVIDER=none`에서도 공통 계획과 규칙 렌더러로
   다양화가 적용되어야 한다.
6. **기존 API 계약을 보존한다.** `answer`, `reply`, `documents`, `benefit`,
   `eligibility`, `apply_detail`, `suggested_actions` 등 현재 응답 필드를 제거하거나
   의미를 바꾸지 않는다.

## 5. 아키텍처

새 모듈 `ai/response_planner.py`가 상담 답변의 내용이 아니라 **표현 계획**을 만든다.

```text
질문 + 선택 정책 + 정책 사실 + 사용자 조건 + 최근 대화
                         |
                         v
                  ResponsePlanner
                         |
                         v
        focus / sections / detail / tone variant
        repetition mode / follow-up / next action
                         |
             +-----------+-----------+
             |                       |
             v                       v
       규칙 기반 렌더러          LLM 프롬프트 구성
             |                       |
             +-----------+-----------+
                         |
                         v
                    최종 상담 답변
```

### 5.1 `ResponsePlanner`

`ResponsePlanner`는 LLM을 호출하지 않는 결정론적 컴포넌트다. 입력으로 다음 값을 받는다.

- `policy_key`: `doc_id` 또는 `source_table:source_id`
- `intent`: `docs`, `benefit`, `eligibility`, `apply_how`, `overview` 등
- `question`: 현재 사용자 질문
- `policy_facts`: 기존 로더와 계산기가 만든 구조화 사실
- `user_context`: 나이, 지역, 소득 등 제공된 사용자 조건
- `conversation_context`: 최근 사용자·상담 Agent 턴과 이미 다룬 의도

출력은 다음 형태의 `ResponsePlan`이다.

```python
{
    "focus": "deadline_risk",
    "section_order": ["summary", "period", "documents", "next_step"],
    "detail_level": "standard",
    "opening_variant": "direct",
    "repetition_mode": "fresh",
    "follow_up_kind": "missing_user_condition",
    "suggested_action_order": ["eligibility", "docs", "create_apply_plan"],
}
```

계획에는 정책 사실의 실제 문장이나 새로운 수치가 들어가지 않는다.

### 5.2 맥락 분류

계획기는 다음 신호를 조합한다.

- **질문 의도:** 서류, 혜택, 자격, 신청 방법
- **구체성:** 넓은 질문인지 특정 금액·서류·기간을 묻는 질문인지
- **사용자 조건 완성도:** 개인화 판정에 필요한 정보가 충분한지
- **대화 단계:** 최초 설명, 보충 설명, 반복 확인, 신청 준비 전환
- **이전 답변:** 직전 턴에서 이미 강조한 섹션과 후속 질문
- **정책 특성:** 현금 지원, 대출, 주거 공고, 훈련, 창업 공고 등

예시는 다음과 같다.

| 맥락 | 계획 결과 |
|---|---|
| 처음 “무슨 혜택이야?” | 핵심 혜택 → 금액/기간 → 확인 사항 |
| 혜택 설명 뒤 “그래서 총 얼마야?” | 계산 결과를 첫 문장에 배치하고 부가 설명 축소 |
| 프로필 없이 “내가 가능해?” | 부족한 사용자 조건을 먼저 안내 |
| 서류 안내 후 “어디서 발급해?” | 발급처와 링크를 먼저 배치 |
| 동일 내용을 바로 반복 질문 | 앞 답변을 장황하게 반복하지 않고 핵심 확인형으로 답변 |
| 신청 의사가 드러난 질문 | 신청 기간과 바로 할 다음 행동을 우선 배치 |

### 5.3 결정론적 변형 선택

`opening_variant`와 동일 수준의 표현 변형은 다음 입력으로 만든 안정적인 키를 사용한다.

```text
policy_key + intent + conversation_phase + focus + detail_level
```

이 키를 정규화한 후 준비된 변형 집합의 인덱스로 매핑한다. 난수를 사용하지 않으므로 같은
맥락에서는 같은 계획과 규칙 기반 변형을 선택한다. LLM 경로에서는 문구가 조금 달라질 수
있지만 초점, 구조, 사실, 후속 질문 종류는 같은 계획을 따른다. 대화 단계나 초점이 달라질
때만 계획 자체가 달라진다.

### 5.4 반복 방지

최근 상담 Agent 턴에서 사용한 다음 정보를 `conversation_context`로 추출한다.

- 이미 설명한 의도
- 직전 `focus`
- 직전 섹션 순서
- 직전 후속 질문 종류

`/agent/converse`는 이 정보를 대화 턴의 내부 payload에 `response_plan_meta`로 저장한다.
이 메타데이터는 공개 API 응답에 추가하지 않는다. `/chat`은 클라이언트가 전달한 이전
사용자 메시지의 의도 순서와 현재 질문의 구체성을 이용해 같은 맥락 정보를 재구성한다.

같은 의도가 연속되면 `repetition_mode`를 다음 중 하나로 정한다.

- `fresh`: 처음 설명
- `clarify`: 더 구체적인 질문에 대한 보충
- `confirm`: 동일 질문 반복에 대한 핵심 재확인
- `deepen`: 이전에 다루지 않은 세부 정보 추가

반복 방지는 DB에 없는 내용을 추가하는 방식이 아니라, 이미 확인된 사실의 설명 밀도와
순서를 바꾸는 방식으로만 동작한다.

## 6. API별 데이터 흐름

### 6.1 `/chat`

1. `PolicyChatAgent.answer()`가 현재처럼 정책 컨텍스트와 메시지를 정규화한다.
2. 최근 `messages`에서 대화 맥락을 추출한다.
3. `ResponsePlanner.plan()`을 호출한다.
4. LLM이 활성화된 경우:
   - 기존 grounded 정책 컨텍스트와 `ResponsePlan`을 함께 시스템 프롬프트에 제공한다.
   - LLM은 계획의 초점과 순서를 따르되 DB 사실만 사용한다.
5. LLM이 비활성화되거나 실패한 경우:
   - 같은 `ResponsePlan`을 규칙 기반 렌더러에 전달한다.
6. 기존 `ChatResponse` 형식을 그대로 반환한다.

### 6.2 `/agent/converse`

1. 추천 이후 선택된 정책에 대한 의도를 기존 `IntentRouter`가 분류한다.
2. `backend/main.py`가 현재 세션의 최근 턴을 읽어 `ConverseAgent.respond()`에 전달한다.
3. `ConverseAgent`는 현재처럼 `ApplyAgent`, `BenefitEstimator`,
   `PolicyChatAgent`에서 사실 데이터를 얻는다.
4. `ResponsePlanner.plan()`이 공통 표현 계획을 만든다.
5. 의도별 규칙 렌더러가 계획에 맞춰 제목, 섹션 순서, 설명 깊이, 후속 행동을 조립한다.
6. 기존 구조화 필드와 세션 저장 방식을 유지한다.

`NEED_RECOMMENDATION`, 정책 미선택 안내, 단순 선택 확인은 이 계획기의 범위에서 제외한다.

## 7. 렌더링 전략

### 7.1 규칙 기반 렌더러

각 의도는 사실을 생성하는 함수와 문장을 조립하는 함수를 분리한다.

- `docs`: 제출 서류, 발급처, 신청 링크
- `benefit`: 계산된 지원 형태, 월액, 총액, 기간
- `eligibility`: 판정 결과와 근거, 추가 확인 조건
- `apply_how`: 방법, 기간, 링크, 문의처

렌더러는 `ResponsePlan`에 따라 다음만 변경한다.

- 도입 문장
- 사실 섹션 순서
- 표시할 보조 섹션
- bullet의 설명 길이
- 마지막 후속 질문 또는 행동 제안

구조화 응답 필드의 값은 기존 계산 결과를 그대로 사용한다.

### 7.2 LLM 렌더러

LLM 프롬프트에는 `ResponsePlan`을 명시적으로 추가한다.

- `focus`를 첫 문단에서 다룬다.
- `section_order` 순서로 답한다.
- `detail_level`에 맞는 길이를 지킨다.
- `repetition_mode`에 따라 반복 정도를 조절한다.
- `follow_up_kind`에 맞는 질문 하나만 던진다.
- 제공되지 않은 사실을 만들지 않는다.

LLM의 sampling 값을 높이는 것은 핵심 해결책으로 사용하지 않는다. 공급자별 기본 생성은
현재처럼 재현성을 우선하고, 다양성은 계획 차이에서 만든다. 필요하면 자유 텍스트 상담에만
낮은 수준의 configurable sampling을 후속 옵션으로 추가할 수 있으나 이번 범위에는 포함하지
않는다.

## 8. 후속 질문 선택

후속 질문은 일반 문장 목록을 무작위 선택하지 않고 현재 부족한 정보와 다음 업무 단계에서
결정한다.

우선순위는 다음과 같다.

1. 개인화 판정에 필요한 사용자 정보
2. 신청 마감 또는 접수 가능 여부
3. 아직 설명하지 않은 핵심 의도
4. 신청 준비 플랜 생성
5. 더 확인할 내용이 없으면 공식 링크 확인 안내

이미 직전 답변에서 사용한 후속 질문 종류는 가능한 경우 반복하지 않는다.

## 9. 오류 및 폴백 처리

- 계획기 입력이 일부 비어 있어도 기본 계획을 반환한다.
- 대화 이력을 읽지 못하면 현재 질문과 정책 사실만으로 `fresh` 계획을 만든다.
- 알 수 없는 의도는 기존 `overview` 또는 `unclear` 처리로 돌아간다.
- LLM 호출 실패 시 동일한 `ResponsePlan`을 규칙 렌더러가 사용한다.
- DB에 없는 사실은 기존 정책대로 “DB에 없음/공식 공고 확인 필요”로 표시한다.
- 계획기 오류가 상담 API 전체의 500 오류로 이어지지 않도록 안전한 기본 계획을 둔다.

### 상담 Agent 회귀 테스트

- `/chat`의 LLM-off 답변이 동일 사실을 유지하면서 맥락별 구조를 달리한다.

## 11. 수용 기준

- 추천 결과와 추천 순위는 변경되지 않는다.
- 정책을 선택한 이후의 `/chat`과 `/agent/converse`에만 적용된다.
- 두 API가 공통 `ResponsePlanner`를 사용한다.
- `LLM_PROVIDER=none`에서도 맥락 기반 구조·강조점·후속 질문 변화가 나타난다.
- 같은 정책, 의도, 사용자 조건, 대화 단계에서는 계획·구조·사실이 안정적이다.
- 규칙 기반 답변은 같은 맥락에서 동일하고, LLM 답변은 문구가 조금 달라도 같은 계획을 따른다.
- 대화 단계나 추가 정보가 바뀌면 답변 초점 또는 구조가 자연스럽게 달라진다.
- 정책 수치, 날짜, 자격, URL은 기존 DB·규칙 결과에서만 나온다.
- 기존 API 응답 스키마와 추천·신청 워크플로우 회귀 테스트가 통과한다.

## 12. 비목표

- 응답을 매 호출마다 무조건 다르게 만드는 것
- temperature를 높여 무작위 표현을 만드는 것
- 정책 DB 밖의 상식이나 웹 정보로 답변을 확장하는 것
- 추천 정책의 다양성 또는 검색 품질을 변경하는 것
- 상담 Agent가 새로운 추천 목록을 생성하게 만드는 것
